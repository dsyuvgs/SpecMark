# PRD自检规则库

## 〇、评判标准

> PRD审查不像分类任务有明确的"对/错"，但必须有可量化的评判方式。
> 以下标准参考了 ISTQB 测试标准中的缺陷分类体系、IEEE 830 软件需求规格标准、
> 以及软件需求评审行业实践和 PRD质量评估标准。

### 1. 核心指标

PRD审查的质量用三个指标衡量：

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **检出率（Recall）** | 应该发现的问题中，实际发现了多少 | 检出问题数 / 应检出问题数 × 100% |
| **误报率（False Positive Rate）** | 报出来的问题里，有多少不是真问题 | 误报数 / 总报告数 × 100% |
| **准确率（Precision）** | 报出来的问题里，有多少是真问题 | 真问题数 / 总报告数 × 100% |

三者关系：准确率 = 1 - 误报率。检出率和准确率往往互相矛盾——报得多容易误报，报得少容易漏报。

### 2. 行业基准线

| 指标 | 行业及格线 | 行业优秀线 | 我们的目标 | 依据 |
|------|-----------|-----------|-----------|------|
| 检出率 | ≥70% | ≥90% | ≥80% | 需求评审实践：人工评审检出率通常60-80%（CSDN需求评审标准），Agent应超过人工 |
| 准确率 | ≥75% | ≥90% | ≥85% | 误报太多会降低信任度，准确率必须够高 |
| 误报率 | ≤25% | ≤10% | ≤15% | 误报率>25%时用户会认为Agent不可靠 |
| 严重问题检出率 | ≥90% | ≥98% | ≥95% | P1场景错配必须几乎不漏，这是最严重的风险 |

### 3. "应检出问题"怎么定

PRD审查的难点是：没有标准答案。所以"应检出问题"必须人工定义：

| 步骤 | 方法 | 说明 |
|------|------|------|
| 1. 准备测试PRD | 写一份故意埋了已知问题的PRD | 包含P1-P5各类问题 |
| 2. 人工标注问题 | 至少1人逐条标注PRD中的所有问题 | 按P1-P5维度标注 |
| 3. 跑Agent | 把同一份PRD输入Agent | 记录Agent的输出 |
| 4. 对比 | Agent检出 vs 人工标注 | 计算检出率、误报率、准确率 |

### 4. 测试PRD的设计要求

| 要求 | 说明 | 示例 |
|------|------|------|
| 覆盖所有维度 | P1-P5每个维度至少1个问题 | 场景错配+埋点违规+数据无来源+缺异常处理+验收模糊 |
| 包含无问题段落 | 至少30%内容是正常的 | 防止Agent"什么都报" |
| 包含边界case | 灰名单术语、短PRD、草稿片段 | 验证灰名单处理和短PRD降级逻辑 |
| 包含不同行业 | 至少2个行业的PRD | 验证分行业术语适配 |

### 5. 什么算"真问题"vs"误报"

| 判定 | 条件 | 示例 |
|------|------|------|
| 真问题 | Agent报的问题确实存在，且风险等级正确 | 报"多租户→场景错配P1"，PRD里确实写了多租户 |
| 误报 | Agent报的问题不存在，或风险等级过高 | 报"缺少异常处理P3"，但PRD里实际已写了异常处理 |
| 等级偏差 | 问题存在但等级不对 | P1报成P3，或P3报成P1。等级偏差不算误报，但记录 |
| 漏报 | 应检出但没检出 | PRD里写了"RBAC权限管理"但Agent没报P1 |

### 6. 对比基线

| 对比对象 | 怎么比 | 合格标准 |
|---------|--------|---------|
| 人工评审 | 同一份PRD，1个PM评审 vs Agent | Agent检出率 ≥ 人工检出率 |
| 裸模型 | 同一份PRD，纯LLM检查 vs Agent | Agent准确率 > 裸模型 +15个百分点（裸模型容易漏检和误报） |
| 行业平均 | 人工需求评审平均检出率60-80% | Agent检出率 > 80% |

### 7. 测试方法

| 方法 | 数据量 | 频率 | 说明 |
|------|--------|------|------|
| 标准PRD测试 | 1份埋了10+问题的PRD | 每次改完都跑 | 计算检出率、误报率 |
| 真实PRD测试 | 3份真实PRD | 每个版本1次 | 验证实际使用效果 |
| 跨行业测试 | 至少2个行业的PRD | 发版前 | 验证术语适配的通用性 |
| 人工抽检 | 随机5份Agent报告 | 每月 | 人工复核Agent的判定 |

### 8. PRD审查的特殊性

PRD审查和反馈分类不一样，有几个特殊性要说明：

| 特殊性 | 说明 | 对评判的影响 |
|--------|------|------------|
| 没有绝对正确答案 | 同一段PRD，不同PM可能看法不同 | 检出率比准确率更重要——宁可多报不可漏报 |
| 风险等级有主观性 | P2还是P3可能见仁见智 | 等级偏差不算误报，但记录 |
| 行业差异大 | 电商PRD和AI平台PRD关注点完全不同 | 必须分行业测试 |
| 短PRD信息不足 | 一两句话的PRD很难做完整审查 | 短PRD的检出率基准线降低10% |

## 一、埋点方案怎么设计

### 核心思路：通用事件+动态参数

埋点的核心矛盾：事件数量有限 vs 用户行为无限。做法是用少量通用事件配合动态参数，不是一个行为一个事件。

### 通用事件设计

| 通用事件 | 覆盖场景 | 核心参数 | 参数枚举示例 |
|----------|---------|---------|-------------|
| page_view | 所有页面浏览 | page_name, stay_duration, from_source | 首页/商品详情/购物车/订单/个人中心 |
| item_interact | 所有与内容对象的交互 | item_id, interact_type, module_id | 点击/收藏/加购/分享/评价/对比 |
| click_action | 所有可交互元素的点击 | click_type, target_id, target_position | 按钮/Tab/弹窗/浮层/快捷入口 |
| cart_operate | 购物车所有操作 | operate_type, item_id, quantity | 加购/减购/修改数量/删除/清空 |
| order_action | 订单全流程 | order_id, action_type, amount | 创建/取消/修改/查看/催单 |
| payment_action | 支付相关 | payment_type, amount, status | 微信/支付宝/余额/分期/代付 |
| user_action | 用户身份与属性操作 | action_type, field_name | 注册/登录/授权/修改资料/设置偏好 |
| share_action | 分享传播 | share_channel, content_type | 微信好友/朋友圈/群聊/复制链接 |
| search_action | 搜索行为 | keyword, result_count, click_position | 搜索/筛选/排序/历史搜索 |
| error_exception | 异常报错 | error_code, error_page, api_url | 接口失败/支付失败/加载超时 |

### 公共参数（所有事件默认带）

| 参数 | 说明 |
|------|------|
| uid | 用户唯一ID（没登录就是设备ID） |
| did | 设备唯一ID |
| ts | 时间戳 |
| platform | 平台（iOS/Android/小程序/Web） |
| version | 应用版本号 |
| city | 用户城市 |
| community_id | 社区团购场景下的团长/小区ID |

### 设计步骤

1. 梳理核心用户路径（注册→浏览→加购→下单→支付→分享）
2. 为每个路径节点选通用事件
3. 用参数枚举区分具体行为
4. 设计漏斗指标（加购率=加购用户/浏览用户，下单转化率=下单用户/加购用户，支付成功率=支付成功/提交订单）
5. 预留扩展参数（ext字段），避免后续新增行为时还要加事件

### 埋点数量受限时的优先级

1. 支付成功（最核心的商业数据）
2. 订单创建（转化漏斗关键节点）
3. 加入购物车（购买意向）
4. 页面浏览（流量分析基础）
5. 商品交互（内容分析基础）
6. 分享行为（增长分析）
7. 搜索行为（需求分析）
8. 用户注册/登录（用户分析）
9. 异常报错（稳定性监控）
10. 其他行为

## 二、PRD怎么写

### 核心模块与验收标准写法

| 模块 | 必须包含 | 常见遗漏 |
|------|---------|---------|
| 产品概述 | 目标用户、核心场景、价值主张 | 缺场景描述 |
| 功能需求 | 用户故事+功能描述+验收标准 | 验收标准写成了功能描述 |
| 业务规则 | 边界条件、异常处理、权限控制 | 只写正常流程不写异常 |
| 非功能需求 | 性能、安全、兼容性 | 完全忽略 |
| 整体验收标准 | 上线标准、灰度策略 | 没有可量化的标准 |

### 验收标准写法公式

正确：在[条件]下，执行[操作]，预期[结果]
错误：功能正常使用（太模糊）

示例：
- ✅ 在用户未登录状态下点击"收藏"，弹出登录引导弹窗，点击"登录"跳转登录页
- ❌ 收藏功能正常

### 用户故事拆分原则

- 一个用户故事只描述一个完整行为
- 按INVEST原则：Independent、Negotiable、Valuable、Estimable、Small、Testable
- 过大的故事必须拆分（如"用户管理"→"注册""登录""修改资料""注销"）

### ToC产品禁用模块

以下术语/模块在ToC产品PRD里不能出现：多租户、SSO单点登录、RBAC权限管理、审批流、部门管理、组织架构、角色权限。

## 三-B、企业级术语灰名单（ToC产品需人工确认）

| 术语 | 灰名单等级 | 用户端替代建议 | 技术端可保留条件 |
|------|-----------|--------------|----------------|
| Webhook | ⚠️🟡中 | 改为消息通知/实时提醒 | 仅后台技术文档使用，用户界面不出现 |
| OAuth2.0 | ⚠️🟡中 | 改为第三方登录/一键登录 | 仅技术架构文档使用，用户界面不出现 |
| API | ⚠️🟡中 | 改为接口/服务 | 仅技术文档使用 |
| CDN | ⚠️🟡中 | 删除（用户无需知晓） | 仅运维文档使用 |
| Redis/MySQL | ⚠️🟡中 | 删除（用户无需知晓） | 仅技术架构文档使用 |
| 微服务 | ⚠️🟡中 | 删除（用户无需知晓） | 仅技术架构文档使用 |

判定规则：
- 若术语出现在用户可见的PRD功能描述中 → 标注「⚠️ 该术语未收录，请人工确认」，等级🟡中
- 若术语仅出现在技术架构/后台描述中 → 标注「⚠️ 该术语为技术实现术语，用户端建议替换」，等级🟢低
- 若术语与黑名单术语同时出现（如"多租户+Webhook"）→ 黑名单术语🔴高 + 灰名单术语🟡中，分别列出

## 四、文档类型判定规则

| 文档类型 | 判定特征 | 检查策略 |
|----------|---------|---------|
| 完整PRD | 包含产品概述+功能需求+业务规则+验收标准 | 全量检查，所有维度正常评级 |
| 草稿片段 | 仅包含功能描述，缺少验收标准/业务规则 | 非核心检查项风险等级自动降一级，但P3缺少异常处理仍必须检查 |
| 需求备注 | 一两句话描述需求 | 至少检查P1场景错配，其他维度标注「文档不完整，建议补充后检查」 |

### 短PRD异常处理检查规则

即使PRD只有1-2句话描述正常流程，也必须检查P3缺少异常处理：
- 若仅描述正常流程（如"用户点击X后，系统执行Y"），未提及任何异常/失败/边界情况，应标记P3缺少异常处理
- 风险等级：短PRD为🟡中，完整PRD缺失异常处理为🔴高
