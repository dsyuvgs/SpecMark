## 能帮你做什么？

### ✅ 省90%的大模型API费用
92.6%的反馈走规则引擎直接出结果，不需要调用大模型，零token消耗。只有7.4%的复杂case才需要LLM介入。

### ✅ 输出永远稳定
同样的输入永远得到同样的结果。不会出现"这次分类对、下次分类错"的情况，因为确定性决策由代码规则引擎完成。

### ✅ 每条结果都可追溯
每条分类结果都标注了来源——是"铁律拦截"还是"LLM分类"，你可以追溯到具体是哪条规则做出的判断。

### ✅ 批量处理速度10倍+
1000条反馈分类，纯代码处理只需2分钟。裸LLM逐条调用需要20分钟+。

## 核心架构

把"理解"和"决策"分开了——LLM负责理解语义，代码负责做确定性决策：

```
用户输入（文字 + 可选文件）
     │
     ▼
[LLM+知识库] 轻量分类器 —— 先让LLM理解用户想干什么
     │
     ▼
[代码] 路由验证 —— 再用代码二次确认，防止LLM乱判
     │
     ├── PRD审查（单条/批量） → 知识库检索 → LLM检查 → Jira格式输出
     ├── 反馈分类（单条）     → 知识库检索 → LLM特征提取 → 代码规则引擎 → 输出
     ├── 反馈分类（批量）     → 纯代码关键词匹配 → 汇总输出
     ├── 场景声明             → 记录场景 → 后续检查更准
     └── 无效输入             → 拒绝
```

**为什么这样分：**
- LLM擅长理解语义（反讽、隐含需求），关键词搞不定
- 做决策必须确定性（"付费用户+核心功能=高优先级"必须每次都一样）
- 路由走错了后面全错，所以用代码做二次校验

- [Dify](https://dify.ai) 账号（免费版即可）

**1. 导入工作流**

把 `workflow/SpecMark最新.yml` 导入 Dify。

| 📚知识库-反馈分类规则 | `knowledge/📚知识库-反馈分类规则.md` |

**3. 绑定知识库**

| 节点名 | 绑定知识库 |
|--------|-----------|
| 📚知识库-PRD规范检查 | 📚知识库-PRD规范检查 |
| 📚知识库-反馈分类规则 | 📚知识库-反馈分类规则 |
| 📚知识库-PRD规范检查(批量) | 📚知识库-PRD规范检查（同一个） |

**4. 配置模型**

使用DeepSeek模型，在Dify里配上API Key。

## FAQ

**Q：支持Coze/其他Agent平台吗？**
A：目前只支持Dify。但核心逻辑（规则引擎+知识库+LLM）是平台无关的，可以迁移到其他支持代码节点的平台。

**Q：怎么自定义分类规则？**
A：修改 `knowledge/📚知识库-反馈分类规则.md` 中的规则和关键词，重新上传到Dify知识库即可。代码节点的铁律规则需要修改 `workflow/SpecMark最新.yml` 中对应节点的代码。

**Q：处理10000条数据会不会崩？**
A：批量分类走纯代码关键词匹配，不调用LLM，稳定性有保障。但Dify单次执行有超时限制，建议分批处理（每批500条）。

**Q：支持中文以外的语言吗？**
A：当前规则引擎的关键词是中文的，暂不支持其他语言。LLM节点本身支持多语言，但铁律拦截只对中文生效。

**Q：为什么不用纯LLM？**
A：三个原因——①成本高（每条都要调API），②不稳定（同样输入可能不同输出），③不可解释（不知道为什么这么分类）。规则引擎解决了这三个问题，LLM只处理规则覆盖不了的复杂case。

**Q：准确率还能更高吗？**
A：可以。优先级准确率目前75%，主要瓶颈是中→高的漏判。改进方向：①扩充核心功能关键词库，②增加品类专属规则，③用更多真实数据训练few-shot样本。
