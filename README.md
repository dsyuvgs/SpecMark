# SpecMark

> 反馈分类 + PRD审查工具。基于规则引擎与LLM的混合架构，大部分场景零API成本，输出稳定可追溯。

## 面向谁？

| 角色 | 场景 | SpecMark做什么 |
|------|------|---------------|
| 产品经理 | 每周手动分类用户反馈耗时多 | 自动分类情绪/类型/优先级，减少重复劳动 |
| AI产品助理 | 需要快速定位高优先级问题 | 付费用户+核心功能自动标高优，降低漏判风险 |
| 运营 | 差评量大，人工处理效率低 | 批量分类，识别反讽和隐含需求 |
| 项目经理 | PRD缺乏规范性审查 | 自动审查PRD规范，输出Jira格式缺陷单 |

## 核心价值

### 低成本
92.6%的反馈由规则引擎直接处理，不调用LLM。仅7.4%的复杂case需要LLM介入，API费用显著降低。

### 输出稳定
确定性决策由代码规则引擎完成，同样输入得到同样输出，避免LLM输出波动导致的分类不一致。

### 可追溯
每条分类结果标注来源——"铁律拦截"或"LLM分类"，可追溯到具体规则，便于审计和调优。

### 批量高效
批量分类走纯代码关键词匹配，不依赖LLM逐条调用，处理速度显著提升。

## 核心架构

规则引擎与LLM分工协作——LLM负责语义理解，代码负责确定性决策：

```
用户输入（文字 + 可选文件）
     │
     ▼
[LLM+知识库] 轻量分类器 —— LLM理解用户意图
     │
     ▼
[代码] 路由验证 —— 代码二次校验，防止LLM误判
     │
     ├── PRD审查（单条/批量） → 知识库检索 → LLM检查 → Jira格式输出
     ├── 反馈分类（单条）     → 知识库检索 → LLM特征提取 → 代码规则引擎 → 输出
     ├── 反馈分类（批量）     → 纯代码关键词匹配 → 汇总输出
     ├── 场景声明             → 记录场景 → 后续检查更准
     └── 无效输入             → 拒绝
```

**设计思路：**
- LLM擅长语义理解（反讽、隐含需求），关键词无法替代
- 确定性决策必须由代码完成（"付费用户+核心功能=高优先级"需要稳定输出）
- 路由是关键节点，代码做二次校验降低误判率

## 快速开始

### 前置条件

- [Dify](https://dify.ai) 账号（免费版即可）
- LLM API Key（见下方模型选择）

### 模型选择

工作流默认使用 DeepSeek，但**支持Dify平台上的任意模型**，导入后在Dify界面中切换即可：

| 模型 | 推荐度 | 说明 |
|------|--------|------|
| DeepSeek | ⭐⭐⭐ | 性价比高，中文理解能力好，默认推荐 |
| 智谱 GLM-4 | ⭐⭐⭐ | 中文能力强，Dify中安装智谱插件后可直接切换 |
| 通义千问 | ⭐⭐ | 阿里云模型，需在Dify中安装对应插件 |
| GPT-4o | ⭐⭐ | 效果好但成本较高，适合对准确率要求极高的场景 |

> 切换方法：导入工作流后，在Dify的节点设置中将模型从 `deepseek-chat` 改为你想用的模型即可。代码节点不依赖模型，无需修改。

### 部署步骤

**1. 导入工作流**

把 `workflow/SpecMark.yml` 导入 Dify。

**2. 创建知识库**

在Dify中创建2个知识库：

| 知识库名称 | 上传文件 |
|-----------|---------|
| 📚知识库-PRD规范检查 | `knowledge/📚知识库-PRD规范检查.md` |
| 📚知识库-反馈分类规则 | `knowledge/📚知识库-反馈分类规则.md` |

**3. 绑定知识库**

| 节点名 | 绑定知识库 |
|--------|-----------|
| 📚知识库-PRD规范检查 | 📚知识库-PRD规范检查 |
| 📚知识库-反馈分类规则 | 📚知识库-反馈分类规则 |
| 📚知识库-PRD规范检查(批量) | 📚知识库-PRD规范检查（同一个） |

**4. 配置模型**

在Dify中配置所选模型的API Key，并将各LLM节点的模型切换为你选择的模型。

**5. 跑测试验证**

```bash
cd scripts
pip install -r requirements.txt
set DIFY_API_KEY=app-xxxx
python batch_test.py
```

## FAQ

**Q：支持Coze/其他Agent平台吗？**
A：目前只支持Dify。核心逻辑（规则引擎+知识库+LLM）是平台无关的，可以迁移到其他支持代码节点的平台。

**Q：支持哪些模型？**
A：支持Dify平台上的任意LLM。默认DeepSeek，智谱GLM-4、通义千问、GPT-4o等均可使用，在Dify界面中切换模型即可。代码节点不依赖具体模型。

**Q：怎么自定义分类规则？**
A：修改 `knowledge/📚知识库-反馈分类规则.md` 中的规则和关键词，重新上传到Dify知识库即可。代码节点的铁律规则需要修改 `workflow/SpecMark.yml` 中对应节点的代码。

**Q：处理大批量数据会不会崩？**
A：批量分类走纯代码关键词匹配，不调用LLM，稳定性较好。但Dify单次执行有超时限制，建议分批处理（每批500条）。

**Q：支持中文以外的语言吗？**
A：当前规则引擎的关键词是中文的，暂不支持其他语言。LLM节点本身支持多语言，但铁律拦截只对中文生效。

**Q：为什么不用纯LLM？**
A：三个原因——①成本高（每条都要调API），②不稳定（同样输入可能不同输出），③不可解释（不知道为什么这么分类）。规则引擎解决了这三个问题，LLM只处理规则覆盖不了的复杂case。

**Q：优先级准确率还能提升吗？**
A：可以。优先级准确率目前约75%，主要瓶颈是中→高的漏判。改进方向：①扩充核心功能关键词库，②增加品类专属规则，③用更多真实数据优化few-shot样本。
