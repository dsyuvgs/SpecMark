## 一、项目是什么？解决什么问题？

### 1.1 一句话介绍

**SpecMark** 是一个在 Dify 上搭的工作流 Agent，帮产品经理做两件事：
1. **PRD 审查**：把产品需求文档丢进去，自动检查有没有规范风险，输出 Jira 格式的缺陷单
2. **数据标注**：把用户反馈（评价、投诉）丢进去，自动分类标注（情绪/类型/优先级/需求），省去人工逐条标注

### 1.2 为什么要做这个？

产品经理日常有两个痛点：
- **PRD 写完没人审**：小团队没有专门的评审流程，PRD 里埋了坑（验收标准模糊、缺少异常处理）到开发阶段才发现，返工成本高
- **用户反馈太多标不过来**：运营丢过来几百条差评，PM 手动一条条分类、标优先级，效率极低

SpecMark 就是把这两件重复劳动自动化。

### 1.3 技术栈

| 技术 | 用途 | 为什么选它 |
|------|------|-----------|
| **Dify** | 工作流编排平台 | 可视化搭工作流，不用写后端代码 |
| **DeepSeek Chat** | 大语言模型（LLM） | 便宜，中文理解能力够用 |
| **YAML** | 工作流配置文件格式 | Dify 用的就是这个格式 |
| **Python（代码节点）** | 规则引擎、数据处理 | Dify 内置的代码执行环境 |

## 二、整体架构

### 2.1 架构图

```
用户输入（文字 + 可选文件）
     │
     ▼
┌─────────────┐
│ 轻量分类器    │  ← LLM：判断用户想干什么
│ (LLM节点)    │
└──────┬──────┘
       │ 输出：{"type": "prd_check", "confidence": 0.9}
       ▼
┌─────────────┐
│ 路由验证      │  ← 代码节点：二次校验 + 文件检测
│ (Code节点)    │     合并为6种最终类型
└──────┬──────┘
       │ 输出：validated_type = "prd_review_file" 等
       ▼
┌─────────────┐
│ 条件分支      │  ← if-else：根据类型走不同路径
│ (If-Else节点) │
└──────┬──────┘
       │
       ├── prd_review_file  → PRD文档提取 → PRD批量格式化 → PRD批量检查 → PRD批量风险计数 → 结果
       ├── prd_review_text  → PRD风险检查(+读者确认) → 风险计数与Jira格式 → 结果
       ├── data_label_file  → 反馈文档提取 → 批量分类 → 批量结果汇总 → 结果
       ├── data_label_text  → 反馈特征提取(LLM) → 反馈规则引擎(代码) → 结果
       ├── scene_declare    → 场景识别 → 场景JSON解析 → 场景确认回复
       └── invalid_input    → 无效输入回复
```

### 2.2 为什么是"两层路由"而不是一层？

意图识别分了两步：**LLM分类器 → 代码路由验证**。为什么不直接让LLM一步到位？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 只用LLM | 简单 | LLM不稳定，可能输出格式错误、漏判文件 |
| 只用代码 | 稳定 | 代码理解不了自然语言的语义 |
| **LLM + 代码** | LLM负责语义理解，代码负责规则兜底 | 多一个节点 |

LLM 擅长理解"这段话是什么意思"，但不擅长精确判断"有没有上传文件"。代码擅长精确判断，但不擅长理解语义。两层配合，准确率最高。

## 三、6条路径详解

### 3.1 PRD审查（有文件）— 批量检查

**触发条件**：用户上传了文件 + 文字中包含PRD相关意图（"检查"/"PRD"/"审查"等）

**流程**：
```
用户上传 prd_test_cases.md + 输入"检查"
  → PRD文档提取：从文件中提取纯文本
  → PRD批量格式化：把文本拆成多条PRD片段，编号格式化
  → PRD批量检查：LLM逐条检查每段PRD的风险
  → PRD批量风险计数：统计风险数量，格式化为Jira格式
  → 输出结果
```

关键点：
- **PRD批量格式化**（代码节点）：把文档内容按段落/条目拆开，每条加上编号 `[PRD-1]`、`[PRD-2]`...，这样LLM能逐条检查，不会混淆
- **PRD批量风险计数**（代码节点）：解析LLM输出，统计🔴高/🟡中/🟢低各几个，格式化为Jira缺陷单。加了异常兜底，LLM输出格式不对也不会崩

### 3.2 PRD审查（无文件）— 单条检查

**触发条件**：用户直接粘贴PRD文本，没有上传文件

**流程**：
```
用户输入 "用户登录功能：输入手机号获取验证码即可登录"
  → PRD风险检查：LLM检查 + 读者确认提示
  → 风险计数与Jira格式：统计风险，格式化输出
  → 输出结果
```

关键点：
- **读者确认**：如果用户还没声明"这个PRD是给谁看的"（老板/开发/测试），LLM会在输出开头加提醒。不同读者关注点不同：老板看商业逻辑，开发看边界条件，测试看异常场景
- 这是产品亮点——不是所有PRD审查工具都会考虑读者视角

### 3.3 数据标注（有文件）— 批量分类

**触发条件**：用户上传了文件 + 文字中包含标注意图（"分类"/"标注"/"这些反馈"等）

**流程**：
```
用户上传 feedback.csv + 输入"帮我分类"
  → 反馈文档提取：从文件中提取纯文本
  → 批量分类：代码节点，逐行解析+关键词匹配+规则引擎
  → 批量结果汇总：汇总所有行的分类结果
  → 输出结果
```

关键点：
- **批量分类完全用代码**，不用LLM——100条反馈全走LLM太慢太贵，结果还不稳定
- 支持 CSV/JSON/TXT/MD 等格式，自动检测分隔符

### 3.4 数据标注（无文件）— 单条分类

**触发条件**：用户直接粘贴一条反馈文本

**流程**：
```
用户输入 "扫地机器人噪音太大，晚上被吵醒了"
  → 反馈特征提取：LLM提取结构化特征（情绪/类型/关键词）
  → 反馈规则引擎：代码节点，基于特征做最终分类
  → 输出结果
```

关键点：
- **为什么单条走LLM + 代码，批量只走代码？** 单条反馈信息量少，纯关键词匹配不够准，需要LLM理解语义。批量数据量大，关键词匹配已经够用
- **LLM只负责"提取特征"**（输出JSON），**代码负责"做决策"**（输出最终分类）——决策逻辑可解释、可调试

### 3.5 场景声明

**触发条件**：用户输入包含行业/场景声明（"我是做电商的"/"给开发看"）

**流程**：
```
用户输入 "我是做电商的，给开发看"
  → 场景识别：LLM提取场景和读者类型
  → 场景JSON解析与更新：代码节点，解析JSON，匹配SCENE_CORE_MAP
  → 场景确认回复：确认已记录
```

关键点：
- **SCENE_CORE_MAP**：51个品类的核心功能映射表。声明场景后，后续PRD检查和反馈分类都能用对应品类的核心功能列表，提高准确率
- 场景信息存在**会话变量**里，整个对话期间持续有效

### 3.6 无效输入

**触发条件**：输入既不像PRD也不像反馈，也不像场景声明

**输出**：⚠️ 请输入有效的待检查内容。你可以粘贴PRD片段或用户反馈让我检查，也可以告诉我你的行业类型。

## 四、核心专业名词解释

### 4.1 平台与框架

| 名词 | 全称 | 解释 |
|------|------|------|
| **Dify** | — | 开源的 AI 应用开发平台，提供可视化工作流编排，不用写代码就能搭 Agent |
| **工作流（Workflow）** | — | Dify 中的核心概念，把一个复杂任务拆成多个节点，按顺序/条件执行 |
| **DSL** | Domain Specific Language | 领域特定语言。Dify 的工作流配置文件（YAML格式）就是一种 DSL |
| **Agent** | — | 能自主完成任务的 AI 程序。用户说一句话，它自动判断意图、选择路径、执行任务 |
| **LLM** | Large Language Model | 大语言模型。本项目用的是 DeepSeek Chat，负责理解自然语言、生成文本 |
| **YAML** | YAML Ain't Markup Language | 一种配置文件格式，用缩进表示层级关系。Dify 的工作流就是 YAML 文件 |

### 4.2 工作流节点类型

| 名词 | 解释 | 本项目中的使用 |
|------|------|---------------|
| **Start 节点** | 工作流入口，接收用户输入 | 用户输入节点 |
| **LLM 节点** | 调用大语言模型，输入prompt，输出文本 | 轻量分类器、PRD风险检查、反馈特征提取等 |
| **Code 节点** | 执行 Python 代码 | 路由验证、反馈规则引擎、批量分类等 |
| **If-Else 节点** | 条件分支，根据变量值走不同路径 | 条件分支节点，6个case |
| **Answer 节点** | 输出最终结果给用户 | 各条路径的输出节点 |
| **Document Extractor 节点** | 从上传文件中提取文本内容 | PRD文档提取、反馈文档提取 |
| **Knowledge Retrieval 节点** | 从知识库中检索相关内容 | 3个知识库节点：PRD规范检查、反馈分类规则、PRD规范检查(批量) |

### 4.3 LLM 相关

| 名词 | 解释 |
|------|------|
| **Prompt** | 给LLM的指令文本，告诉它"你是谁、要做什么、输出什么格式" |
| **System Prompt** | 系统级指令，定义LLM的角色和行为规则 |
| **Temperature** | 温度参数，0=确定性输出（每次结果一样），1=随机性输出。审查类任务用0.1保证稳定 |
| **query_prompt_template** | Dify LLM节点的必填字段，定义用户消息的模板。必须包含 `{{#sys.query#}}`（用户原始输入） |
| **会话变量（Conversation Variable）** | Dify 提供的在同一对话中持续存在的变量。本项目用 `scene` 和 `reader_type` 存储用户声明的场景 |

### 4.4 变量与数据流

| 名词 | 解释 | 示例 |
|------|------|------|
| **变量选择器（Variable Selector）** | Dify 中引用其他节点输出的方式 | `['1776665833871', 'text']` 表示引用某个节点的text输出 |
| **sys.query** | 系统变量，用户的原始文字输入 | 用户输入"检查这个PRD" |
| **sys.files** | 系统变量，用户上传的文件列表 | 用户上传了 prd_test_cases.md |
| **Edge（边）** | 连接两个节点的数据通道 | 条件分支 → PRD风险检查 |

### 4.5 PRD审查相关

| 名词 | 解释 |
|------|------|
| **PRD** | Product Requirements Document，产品需求文档。描述产品要做什么、怎么做、验收标准是什么 |
| **P1 场景错配** | 最高优先级风险。ToC产品出现ToB专属术语（如多租户、SSO、RBAC），说明需求可能写错了场景 |
| **P1 埋点违规** | 最高优先级风险。埋点方案不符合"通用事件+动态参数"的规范 |
| **P2 数据无来源** | 数据类内容没标注出处，无法验证 |
| **P3 缺少异常处理** | 只写了正常流程，没考虑出错时怎么办 |
| **P4 验收标准模糊** | "功能正常运行""体验流畅"这种没法量化验收的表述 |
| **P5 需求不可量化** | "提升用户体验""优化性能"没有具体指标 |
| **Jira 格式** | Jira 是项目管理工具，Jira格式的缺陷单包含：摘要、描述、优先级、标签。SpecMark 输出的就是这个格式，可以直接复制到Jira |
| **灰名单术语** | 技术实现术语（Webhook/OAuth2.0/API/CDN等），不是ToB专属术语，在PRD技术架构描述中出现是合理的，不能判为P1场景错配 |

### 4.6 数据标注相关

| 名词 | 解释 |
|------|------|
| **情绪（Sentiment）** | 反馈的情感倾向：正面/负面/中性 |
| **问题类型** | 反馈属于哪类问题：功能问题/体验问题/性能问题/内容问题/安全问题 |
| **优先级** | 处理优先级：高/中/低。基于多个因素综合判断 |
| **核心功能** | 该品类产品最关键的功能。核心功能出问题 → 高优先级 |
| **SCENE_CORE_MAP** | 51个品类的核心功能映射表。如电商→下单/支付/搜索，游戏→登录/充值/匹配 |
| **GENERIC_CORE** | 通用核心功能列表（崩溃/闪退/支付/登录等），当场景未声明时的回退方案 |
| **隐含需求** | 用户没直接说但暗示想要的功能。如"噪音太大"→ 静音模式，"漏水"→ 防水设计 |
| **NEED_PATTERNS** | 隐含需求提取规则，18条模式匹配。如"每次都要"→ 自动化/记住设置 |
| **FUNC_PROBLEM_KW** | 功能问题关键词列表（69个），匹配到这些词 → 判为功能问题而非体验问题 |
| **读者类型** | PRD的目标读者：老板/开发/测试/设计等。不同读者关注点不同 |

### 4.7 路由相关

| 名词 | 解释 |
|------|------|
| **意图识别** | 判断用户想做什么。是检查PRD？标注数据？还是声明场景？ |
| **路由验证** | 代码节点对LLM分类结果的二次校验。防止LLM误判 |
| **validated_type** | 最终的路由类型，6种：prd_review_file/prd_review_text/data_label_file/data_label_text/scene_declare/invalid_input |
| **条件分支（If-Else）** | 根据 validated_type 的值，走不同的处理路径 |

## 五、关键设计决策（面试必问）

### 5.1 为什么批量分类用代码，不用LLM？

| | 代码方案 | LLM方案 |
|--|---------|---------|
| 100条反馈 | ~2秒 | ~60秒（每条都要调API） |
| 成本 | 几乎为0 | 约0.5-1元（100次API调用） |
| 稳定性 | 同样的输入永远同样的结果 | 可能格式不一致、漏标 |
| 准确率 | 关键词匹配，对明确表达够用 | 语义理解更准 |
| 可解释性 | 每个判断都能追溯到具体关键词 | 黑盒 |

批量场景下，速度和稳定性更重要。关键词匹配对"明确表达"的反馈已经够用。

### 5.2 为什么单条分类用LLM + 代码？

单条反馈信息量少，纯关键词容易误判。比如"这个功能太鸡肋了"——关键词匹配找不到"鸡肋"，但LLM能理解这是负面评价。

所以设计为：**LLM提取特征（语义理解）→ 代码做决策（规则匹配）**。既有语义理解能力，决策逻辑又可解释。

### 5.3 为什么用知识库（RAG）？

| | 纯Prompt方案（旧版） | RAG方案（当前） |
|--|-------------------|---------|
| 准确率 | 63%（规则太长LLM记不住） | 95%（检索相关规则+few-shot） |
| 延迟 | 无检索环节 | 多一次向量检索（<1秒） |
| 维护 | 改YAML重新导入 | 改知识库文件即可，不用重新导入 |
| 稳定性 | 规则太长导致LLM不稳定 | 知识库精准检索，LLM只看相关规则 |
| 适用场景 | 规则量小（<100条） | 规则量大（>100条）或需要few-shot |

早期用纯Prompt方案，规则直接写在YAML里，综合准确率只有63%。原因是规则太多（8条核心规则+30+个few-shot示例），LLM在长prompt中容易遗漏或混淆。改用知识库后，LLM只检索与当前输入最相关的规则和示例，准确率提升到95%。

### 5.4 为什么是两层路由（LLM + 代码）？

LLM擅长理解语义但不稳定，代码擅长精确判断但不理解语义。两层配合：
- LLM：理解"帮我看看这份文档"= 想检查PRD
- 代码：判断有没有上传文件 + 关键词二次确认

单独用任何一层都有盲区，两层互补准确率最高。

### 5.5 为什么PRD批量路径和单条路径要分开？

因为输入格式不同：
- **单条**：用户直接粘贴文本 → 直接给LLM检查
- **批量**：用户上传文件 → 需要先提取文本 → 再拆分格式化 → 再给LLM检查 → 再汇总

批量路径多了"文档提取"、"格式化拆分"、"结果汇总"三个环节，不能复用单条路径。

## 六、数据流详解

### 6.1 单条反馈分类的完整数据流

```
用户输入: "扫地机器人噪音太大，晚上被吵醒了"
    │
    ▼
轻量分类器(LLM):
    输入: sys.query = "扫地机器人噪音太大，晚上被吵醒了"
    输出: {"type": "feedback", "confidence": 0.9}
    │
    ▼
路由验证(Code):
    输入: classifier_output, user_query, has_files
    处理: has_file=False, 无PRD意图, 有负面关键词 → data_label_text
    输出: {"validated_type": "data_label_text", "confidence": 0.85}
    │
    ▼
条件分支(If-Else):
    validated_type contains "data_label_text" → 走反馈特征提取
    │
    ▼
反馈特征提取(LLM):
    输入: sys.query = "扫地机器人噪音太大，晚上被吵醒了"
    Prompt: 包含场景核心功能列表、提取维度说明
    输出: {
      "sentiment": "负面",
      "problem_type": "体验问题",
      "keywords": ["噪音", "吵醒"],
      "severity": "high",
      "is_core_function": true,
      "implied_need": "静音模式/降噪"
    }
    │
    ▼
反馈规则引擎(Code):
    输入: feature_json, original_query, current_scene
    处理:
      1. 解析LLM输出的JSON
      2. 场景匹配: current_scene="扫地机器人" → SCENE_CORE_MAP查到"噪音"是核心功能
      3. 优先级计算: 负面 + 核心功能 + 严重描述 → 高
      4. 隐含需求: NEED_PATTERNS匹配"噪音"→"静音模式/降噪"
    输出: {
      "type": "体验问题",
      "sentiment": "负面",
      "priority": "高",
```

## 八、踩过的坑

### 坑1：100条反馈被数成101条

Dify的文档提取器会在内容前面加一段说明文字，代码没过滤掉，导致多算了一条。后来加了表头清洗逻辑就好了。

### 坑2：PRD批量检查一直报错

Dify的LLM节点有个必填字段叫 `query_prompt_template`，里面必须有 `{{#sys.query#}}`。之前没加，一直报"user message必须包含sys.query"。加上就好了。

### 坑3：LLM节点引用了 sys.files 导致报错

Dify的LLM节点不支持直接引用 `{{#sys.files#}}`，文件内容要通过 Document Extractor 节点提取后才能用。之前好几个LLM节点都写了 `{{#sys.files#}}`，全删掉才好。

### 坑4：知识库检索节点配置问题

早期知识库检索节点用 `retrieval_mode: multiple`，但没配置reranking_model，导致Dify导入报错。改用 `retrieval_mode: single` 后解决。另外，LLM节点的 `context.enabled` 一开始设为false，导致知识库检索结果没传给LLM，Agent等于没有知识库。后来全部改为true并配置了variable_selector。

另一个坑：知识库文件名和节点名不一致，导入后不知道该绑定哪个。后来统一命名：文件名=节点名（📚知识库-PRD规范检查.md → 📚知识库-PRD规范检查节点）。

### 坑5：用户反馈被当成PRD检查了

有人发了一堆差评，Agent把它当PRD去检查了。原因是路由层没区分"用户反馈"和"PRD文档"。后来加了内容类型检测，用关键词判断输入是反馈还是PRD。

### 坑6：风险计数节点没有异常处理

LLM输出格式不对的时候，风险计数节点的 `re.findall` 会返回空列表，或者 `json.loads` 直接报错。后来加了 try-except，输出格式不对也不会崩，会给用户一个友好的提示。

### 坑7：路由逻辑改了好几次

一开始路由只靠LLM判断，结果LLM有时候输出格式不对，有时候高置信度但判错了。后来改成LLM先判断 + 代码二次校验，准确率才上来。

### 坑8：优先级判定太简单

最早只看情绪+类型，没考虑付费用户、缓解词、竞品对比这些特殊情况。后来一个一个加规则，才把准确率提上去。

### 坑9：铁律规则引擎优先级全线输出为"低"

这是项目开发中修复的核心bug。`iron_classify` 函数返回三元组 `(问题类型, 情绪, None)`，优先级写死为 None。三个调用节点拿到 None 后统一给了默认值"低"，导致铁律命中的 185 条结果里优先级全是低。

**修复方案**：
1. 给 `iron_classify` 所有 return 语句补上知识库对齐的优先级（高/中/低）
2. 新增 `apply_priority_rules` 后处理函数，实现四层规则：
   - **缓解词降级**：有"偶尔/有时/还行"等缓解词 → 降一级，但严重问题（闪退/崩溃/食品安全）不降
   - **性能严重词提升**：性能问题 + "太卡/太慢/没法用/受不了" → 升为高
   - **核心功能提升**：功能问题 + 涉及支付/登录/下单等核心功能 → 升为高
   - **体验/内容中度词提升**：体验/内容问题 + 强烈不满表达 → 升为中
3. 扩展 `svc_high_kw` 食安检测关键词：吃出/拉肚子/食物中毒/发霉/馊了/臭了/不新鲜

**效果**：铁律优先级准确率从 0% → 75.1%，高→中误判从 49 条降至 13 条，中→低误判从 21 条降至 4 条。
