# SpecMark Agent 构建SOP

## 概述

本文档记录从0到1构建SpecMark Agent的完整流程，可作为同类项目的标准操作手册。
当前版本：v1.0（2026.05）

---

## 阶段一：需求定义（Day 1）

### 1.1 明确Agent能力边界

| 步骤 | 操作 | 产出 |
|------|------|------|
| 1 | 确定Agent要解决的核心问题 | PRD风险检查 + 用户反馈分类 |
| 2 | 定义输入输出格式 | 输入：文本/文件；输出：结构化JSON |
| 3 | 确定与裸模型的差异点 | 批量处理、知识库、规则引擎、场景适配 |

### 1.2 定义分类维度

| 维度 | 枚举值 | 判定优先级 |
|------|--------|-----------|
| 情绪 | 负向/中性/正向 | 先判 |
| 问题类型 | 功能/性能/体验/内容 | 后判 |
| 优先级 | 高/中/低 | 最后判 |

**关键原则：** 分类维度必须在项目初期就确定，后期修改成本极高。

## 阶段二：数据准备（Day 1-2）

### 2.1 收集原始数据

| 步骤 | 操作 | 注意事项 |
|------|------|---------|
| 1 | 收集200+条真实用户反馈 | 覆盖多个品类（电商/社交/外卖/游戏等） |
| 2 | 每条反馈标注品类 | 品类影响核心功能判定 |
| 3 | 人工标注情绪/类型/优先级 | 严格按规则标注，不要凭感觉 |

### 2.2 生成标准答案（LLM共识法）

**为什么不用纯人工标注？**
- 人工标注成本高（200条×3维度=600次判断）
- 人与人之间一致性差（Cohen's Kappa通常<0.7）
- LLM共识法更高效：3轮投票+仅仲裁争议case

**操作步骤：**

```
1. 编写分类prompt（含规则+few-shot示例）
2. 对每条反馈跑3轮LLM分类
3. 多数投票决定标签（2/3或3/3一致→采纳）
4. 争议case（3轮全不同）→ 人工仲裁
5. 输出ground_truth_consensus.csv
```

**关键脚本：** `generate_consensus_ground_truth.py`

### 2.3 GT质量验证

| 步骤 | 操作 | 目的 |
|------|------|------|
| 1 | LLM共识 vs 原始GT对比 | 找出不一致的case |
| 2 | 逐条审核不一致case | 判断是GT错了还是LLM错了 |
| 3 | 修正GT标签 | 只改确认有误的GT |
| 4 | 重新跑共识测试 | 验证修正后准确率 |

**常见GT错误类型：**
- 建议类标为功能（"希望增加XX功能"→应为体验）
- 正向反馈一律标体验（夸速度→应为性能）
- 服务问题标为功能（配送慢→应为体验）
- 优先级偏低（等40分钟→应为高）

## 阶段三：知识库构建（Day 2）

### 3.1 知识库文件设计

| 知识库 | 内容 | 格式 |
|--------|------|------|
| 反馈分类规则 | 分类规则+易错点+few-shot+品类核心功能映射 | Markdown |
| PRD规范检查 | 风险维度+判定标准+行业参考 | Markdown |

### 3.2 知识库文件编写原则

1. **规则要具体**：不要写"功能失效判功能"，要写"闪退/崩溃/数据丢失/支付失败→功能"
2. **易错点要列举**：每个高频错误都要有"正确分类 vs 错误分类"的对比
3. **Few-shot示例要覆盖**：每个分类维度至少3个正向示例+3个易错示例
4. **品类核心功能要映射**：50+品类的核心功能关键词列表

### 3.3 知识库文件命名

命名必须与工作流中的知识库节点名称一致：
- `📚知识库-反馈分类规则.md` → 对应节点 `📚知识库-反馈分类规则`
- `📚知识库-PRD规范检查.md` → 对应节点 `📚知识库-PRD规范检查`

## 阶段四：工作流搭建（Day 2-3）

### 4.1 Dify工作流结构

```
开始 → 轻量分类器(LLM) → 路由验证(Code) → 条件分支(If-Else)
                                                    │
            ┌──────────┬──────────┬─────────────────┼──────────┐
            ▼          ▼          ▼                 ▼          ▼
        PRD文件    PRD文本    反馈文件           单条反馈    场景声明
            │          │          │                 │          │
            ▼          ▼          ▼                 ▼          ▼
        文档提取    知识库检索  文档提取          知识库检索  场景识别
            │          │          │                 │          │
            ▼          ▼          ▼                 ▼          ▼
        知识库检索   LLM检查    批量格式化        LLM分类    场景确认
            │          │          │                 │          │
            ▼          ▼          ▼                 ▼          ▼
        LLM检查    规则引擎    批量LLM分类       规则引擎    回复
            │          │          │                 │
            ▼          ▼          ▼                 ▼
        规则引擎    结果输出    结果汇总          结果输出
```

### 4.2 关键配置项

| 配置项 | 推荐值 | 原因 |
|--------|--------|------|
| LLM模型 | deepseek-chat（默认）/ 智谱GLM-4 / 通义千问 / GPT-4o | 可插拔，按需选择 |
| temperature | 0.1 | 分类任务需要稳定性 |
| 知识库检索模式 | single | 知识库小，不需要多路召回 |
| context.enabled | true | LLM必须引用知识库检索结果 |

### 4.3 知识库节点配置

```yaml
# 必须配置项
dataset_ids: []          # 导入后手动绑定
retrieval_mode: single   # 单路召回，避免reranking报错
query_variable_selector:
- '开始节点ID'
- sys.query
```

**避坑：** 不要用 `retrieval_mode: multiple`，Dify会要求配置reranking_model，如果留空会报错。

### 4.4 LLM节点context配置

```yaml
context:
  enabled: true
  variable_selector:
  - '知识库节点ID'
  - result
```

**避坑：** `enabled: false` 会导致知识库检索结果不传给LLM，Agent等于没有知识库。

## 阶段五：Prompt工程（Day 3）

### 5.1 Prompt结构

```
1. 角色定义：你是SpecMark反馈分类器
2. 分类维度：情绪/类型/优先级
3. 核心规则：8条必须遵守的规则
4. Few-shot示例：30+个覆盖所有高频错误的示例
5. 输出格式：只输出JSON
```

### 5.2 Few-shot示例选择原则

1. **覆盖所有分类维度**：每个维度至少3个示例
2. **覆盖所有高频错误**：每个已知错误模式至少1个示例
3. **示例要简短**：输入不超过20字，输出只含必要字段
4. **reasoning要精准**：用"="连接因果（如"建议类=体验"）

### 5.3 迭代优化流程

```
1. 初始prompt（只有规则，没有few-shot）→ 跑测试 → 记录错误
2. 分析错误模式 → 按频率排序
3. 对Top错误：先判断GT是否正确 → 修正GT → 再加few-shot
4. 重新跑测试 → 验证提升
5. 重复直到准确率达标
```

**关键原则：** 每次迭代前先想清楚所有问题，不要改一点测一点。

## 阶段六：测试与评估（Day 3-4）

### 6.1 测试方法

```bash
cd scripts
pip install -r requirements.txt
set DIFY_API_KEY=app-xxxx
python batch_test.py
```

### 6.2 评估指标

| 指标 | 及格线 | 优秀线 |
|------|--------|--------|
| 情绪准确率 | ≥85% | ≥92% |
| 类型准确率 | ≥80% | ≥90% |
| 优先级准确率 | ≥75% | ≥88% |
| 综合准确率 | ≥70% | ≥85% |

### 6.3 错误分析

| 步骤 | 操作 |
|------|------|
| 1 | 统计错误维度分布（情绪/类型/优先级各错多少） |
| 2 | 统计错误模式分布（功能→体验、中→高等） |
| 3 | 逐条审核错误，判断GT vs LLM谁对 |
| 4 | 修正GT + 补充few-shot |

## 阶段七：部署上线（Day 4）

### 7.1 Dify部署步骤

| 步骤 | 操作 | 注意事项 |
|------|------|---------|
| 1 | 创建2个知识库 | PRD规范检查 + 反馈分类规则 |
| 2 | 上传知识库文件 | 文件名与节点名一致 |
| 3 | 导入工作流YAML | SpecMark.yml |
| 4 | 绑定知识库 | 3个节点分别绑定对应知识库 |
| 5 | 配置模型API Key | 默认deepseek-chat，可切换为其他模型 |
| 6 | 测试运行 | 验证各链路正常 |
| 7 | 发布上线 | 获取访问链接 |

### 7.2 知识库绑定对应关系

| 工作流节点 | 绑定知识库 |
|-----------|-----------|
| 📚知识库-PRD规范检查 | 📚知识库-PRD规范检查 |

## 常见问题与避坑

| 问题 | 原因 | 解决 |
|------|------|------|
| 导入YAML后知识库节点报错 | dataset_ids为空 | 正常，需手动绑定知识库 |
| reranking_model报错 | retrieval_mode=multiple需要reranking | 改为single |
| Agent分类结果和预期不一致 | LLM没读到知识库 | 检查context.enabled是否为true |
| 综合准确率低 | GT标签本身有误 | 先审GT，再加few-shot |
| 同一反馈每次分类不同 | temperature太高 | 降到0.1 |
| 建议类被判为功能 | prompt没强调"建议类一律体验" | 加few-shot示例 |
| 服务问题被判为功能 | 没区分"软件坏了"vs"服务差" | 加规则：配送/商家/客服→体验 |
