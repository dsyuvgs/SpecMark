# SpecMark

> 产品规范标记工具。给需求规格打上规范标记，不做生成，只做检查和标注。基于Dify平台搭建的工作流Agent。
> 当前版本：v1.0（2026.05）

## 怎么做的

### 把"理解"和"决策"分开了，不让LLM做所有事

```
用户输入（文字 + 可选文件）
     │
     ▼
[LLM+知识库] 轻量分类器 —— 先让LLM理解用户想干什么（接入知识库辅助判断）
     │
     ▼
[代码] 路由验证 —— 再用代码二次确认，防止LLM乱判
     │
     ├── prd_review_file   → 文档提取 → 格式化拆分 → 知识库检索 → 批量PRD检查 → Jira格式输出
     ├── prd_review_text   → 读者确认 → 知识库检索 → 单条PRD检查 → Jira格式输出
     ├── data_label_file   → 文档提取 → 批量分类（纯代码） → 汇总输出
     ├── data_label_text   → 知识库检索 → LLM提取特征 → 代码规则引擎 → 输出
     ├── scene_declare     → 记录场景 → 后续检查更准
     └── invalid_input     → 拒绝
```

为什么这样分：
- LLM擅长理解语义，比如"太棒了，一天闪退八次"是反讽，关键词搞不定
- 但做决策必须确定，"付费用户+核心功能=高优先级"必须每次都一样
- 路由走错了后面全错，所以用代码做二次校验
- 知识库检索让LLM有规则可循，不靠"感觉"分类

### 知识库的作用

| 知识库 | 接入节点 | 作用 |
|--------|---------|------|
| 📚知识库-PRD规范检查 | PRD风险检查 + PRD批量检查 | 提供PRD风险检查规则 |
| 📚知识库-反馈分类规则 | 轻量分类器 + 反馈特征提取 | 提供分类规则和few-shot示例 |

### 批量 vs 单条

| 场景 | 怎么做 | 为什么 |
|------|--------|--------|
| 批量反馈分类 | 纯代码关键词匹配 | 100条走LLM太慢太贵，关键词够用 |
| 单条反馈分类 | 知识库检索 → LLM特征提取 → 代码规则引擎 | 单条信息少，需要LLM理解语义 |
| 批量PRD检查 | 文档提取 → 格式化 → 知识库检索 → LLM逐条检查 | PRD必须LLM理解，但先拆好再查 |
| 单条PRD检查 | 知识库检索 → LLM直接检查 + 读者确认 | 单条直接查，但会问你是给谁看的 |

### Agent vs 裸LLM：核心差异

| 能力 | 裸LLM | SpecMark Agent |
|------|--------|----------------|
| 铁律拦截率 | 0%（每条都消耗token） | **92.6%**（绝大多数零token） |
| 输出确定性 | 每次可能不同 | 同样输入永远同样输出 |
| 可解释性 | 黑盒 | 每条可追溯到具体规则 |
| 成本 | 每条都消耗token | 铁律拦截=零LLM成本 |
| 优先级可调优 | 改prompt，效果不确定 | apply_priority_rules精确控制 |
| 知识库对齐 | 无 | 分类结果与知识库v1.0完全对齐 |
| 批量处理 | 逐条调用 | 纯代码批量，速度10倍+ |

## 怎么用

### 1. 导入工作流

把 [workflow/SpecMark最新.yml](workflow/SpecMark最新.yml) 导入Dify。

### 2. 创建知识库

在Dify中创建2个知识库，分别上传对应文件：

| 知识库名称 | 上传文件 |
|-----------|---------|
| 📚知识库-PRD规范检查 | [knowledge/📚知识库-PRD规范检查.md](knowledge/📚知识库-PRD规范检查.md) |
| 📚知识库-反馈分类规则 | [knowledge/📚知识库-反馈分类规则.md](knowledge/📚知识库-反馈分类规则.md) |

### 3. 绑定知识库

工作流中有3个知识库节点，分别绑定：

| 节点名 | 绑定知识库 |
|--------|-----------|
| 📚知识库-PRD规范检查 | 📚知识库-PRD规范检查 |
| 📚知识库-反馈分类规则 | 📚知识库-反馈分类规则 |
| 📚知识库-PRD规范检查(批量) | 📚知识库-PRD规范检查（同一个） |

### 4. 配置模型

用的DeepSeek模型，在Dify里配上API Key。

### 5. 跑测试

```bash
cd scripts
pip install -r requirements.txt
set DIFY_API_KEY=app-xxxx
python batch_test.py
```

## 用了什么

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| 平台 | Dify | 可视化搭工作流，不用写后端 |
| 模型 | DeepSeek | 便宜，中文理解能力够用 |
| 工作流 | 代码节点 + LLM节点 + 知识库节点 | 代码做确定性决策，LLM做语义理解，知识库提供规则 |
| 测试 | Python + aiohttp | 异步并发跑批量测试 |
| 评估 | LLM共识法 | 3轮投票+争议仲裁，比纯人工标注更高效 |
