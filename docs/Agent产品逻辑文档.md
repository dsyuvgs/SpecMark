# SpecMark Agent 产品逻辑文档

> 当前版本：v1.0（2026.05）

## 一、产品定位

**SpecMark** 是一个产品规范标记工具，核心能力是"给需求规格打上规范标记，不做生成，只做检查和标注"。

### 两大核心功能

| 功能 | 输入 | 输出 |
|------|------|------|
| PRD风险检查 | PRD文档/文本 | 逐项风险标记（等级+修改方向） |
| 用户反馈分类 | 单条/批量反馈 | 情绪+类型+优先级+用户标签 |

### 与裸模型的差异

| 能力 | 裸模型 | SpecMark Agent |
|------|--------|----------------|
| 铁律拦截率 | 0%（每条都消耗token） | **92.6%**（绝大多数零token） |
| 输出确定性 | 每次可能不同 | 同样输入永远同样输出 |
| 可解释性 | 黑盒 | 每条可追溯到具体规则 |
| 成本 | 每条都消耗token | 铁律拦截=零LLM成本 |
| 单条反馈分类 | 能做，但不稳定 | 有规则约束，更稳定 |
| 批量处理500+条 | 做不到 | 能做（工作流自动拆分） |
| PRD风险检查 | 做不到（无领域知识） | 能做（知识库+规则引擎） |
| 结构化输出 | 不稳定 | 稳定（规则引擎兜底） |
| 场景适配 | 做不到 | 能做（51个品类核心功能映射） |
| 优先级可调优 | 改prompt，效果不确定 | apply_priority_rules精确控制 |

## 二、工作流架构

```
用户输入 → 轻量分类器(LLM) → 路由验证(Code) → 条件分支(If-Else)
                                                          │
                    ┌─────────────┬────────────┬───────────┼───────────┬──────────────┐
                    ▼             ▼            ▼           ▼            ▼              ▼
               PRD文件上传   PRD文本输入   反馈文件上传  单条反馈输入  场景声明      无效输入
                    │             │            │           │            │              │
                    ▼             ▼            ▼           ▼            ▼              ▼
              文档提取→知识库  知识库→LLM   文档提取→批量  知识库→LLM  场景识别→确认   回复提示
              →LLM→规则引擎   →规则引擎     分类→汇总    →规则引擎
                    │             │            │           │            │
                    ▼             ▼            ▼           ▼            ▼
               PRD检查结果   PRD检查结果   批量分类报告  分类结果    场景确认回复
```

### 关键节点说明

| 节点 | 类型 | 作用 |
|------|------|------|
| 轻量分类器 | LLM | 快速判断用户意图（PRD检查/反馈分类/场景声明/无效输入） |
| 路由验证 | Code | 解析LLM输出，提取validated_type，处理边界情况 |
| 条件分支 | If-Else | 根据validated_type路由到不同处理链路 |
| 📚知识库-PRD规范检查 | Knowledge | 检索PRD风险检查规则 |
| 📚知识库-反馈分类规则 | Knowledge | 检索反馈分类规则和few-shot示例 |
| 反馈特征提取 | LLM | 基于知识库上下文，对反馈进行三维度分类 |
| 反馈规则引擎 | Code | 解析LLM的JSON输出，兜底修正，格式化结果 |
| PRD风险检查 | LLM | 基于知识库上下文，逐项检查PRD风险 |
| 风险计数与Jira格式 | Code | 统计风险数量，生成Jira格式输出 |

## 三、分类逻辑详解

### 3.1 情绪分类（三分类）

| 情绪 | 判定条件 | 示例 |
|------|---------|------|
| 负向 | 不满/失望/愤怒/功能失效/趋势恶化 | "太垃圾了""支付失败""越来越卡" |
| 中性 | 客观描述/委婉建议/有缓解词 | "能不能加个深色模式""偶尔卡但还行" |
| 正向 | 满意/认可/喜欢 | "超好用""推荐给大家" |

**关键判定规则：**
- 反讽→负向："真厉害，更新后更卡了"
- 功能失效描述→负向：即使语气平淡，"支付失败"本身就是负面
- 有缓解词+整体还行→中性："偶尔有点噪音，但整体还行"
- 趋势恶化→负向："越来越差""越来越贵"

### 3.2 问题类型分类（四分类）

| 类型 | 判定条件 | 核心区分 |
|------|---------|---------|
| 功能 | **软件本身坏了/出错了** | 闪退、崩溃、数据丢失、支付扣钱但订单没生成、页面打不开 |
| 性能 | **物理性能指标差** | 卡顿、慢、耗电、发热、加载慢 |
| 体验 | **软件没坏但服务差/操作差** | 配送慢、商家取消、客服差、太贵、广告多、建议类 |
| 内容 | **算法输出质量差** | 推荐不准、搜索不匹配、翻译错、歌词不同步 |

**最关键的区分：功能 vs 体验**
- 功能 = 软件本身坏了（支付失败、闪退、打不开）
- 体验 = 软件没坏但服务差（配送慢、商家取消、客服不管、太贵了）

**判定顺序：**
1. 先检测功能失效 → 功能
2. 再检测性能问题 → 性能
3. 再检测内容问题 → 内容
4. 最后归体验

**特殊规则：**
- 建议类一律→体验："能不能加个XX""希望增加XX"→体验（即使提到"功能"二字）
- 价格/费用→体验："太贵了""性价比低"→体验
- 算法/推荐/匹配→内容："搜索结果不匹配""推荐不准"→内容
- 正向反馈按维度分类：夸速度→性能，夸功能→功能，夸推荐→内容，夸操作→体验

### 3.3 优先级分类（三分类）

| 优先级 | 触发条件 |
|--------|---------|
| 高 | 严重故障(闪退/崩溃/数据丢失/支付失败) + 付费用户+问题 + 核心功能失效 + 严重影响(等超长时间/被欺骗) + 反讽+问题 |
| 中 | 负向+非核心功能问题 + 负向+一般性能/内容/体验问题 + 中性+功能问题 |
| 低 | 正向情绪 + 中性+非功能问题 + 有明确缓解词的轻微问题 |

**不降级规则：** 闪退、崩溃、数据丢失、支付失败、安全漏洞、隐私泄露——即使有缓解词也不降级。

### 3.4 铁律规则引擎优先级后处理

铁律规则引擎（`iron_classify`）在关键词匹配后，通过 `apply_priority_rules` 函数进行四层后处理：

| 层级 | 规则 | 示例 |
|------|------|------|
| 缓解词降级 | 有"偶尔/有时/还行/勉强"等 → 降一级 | "偶尔卡顿但还行" 高→中 |
| 不降级保护 | 严重问题（闪退/崩溃/食安/支付失败）不降 | "偶尔闪退" 保持高 |
| 性能严重词提升 | 性能+中 + "太卡/没法用/受不了" → 升为高 | "视频卡得没法看" 中→高 |
| 核心功能提升 | 功能+中 + 涉及支付/登录/下单 → 升为高 | "支付一直转圈" 中→高 |
| 体验中度词提升 | 体验+低 + 强烈不满 → 升为中 | "退款7天没处理" 低→中 |
| 内容中度词提升 | 内容+低 + 严重错误 → 升为中 | "商品描述完全不对" 低→中 |

铁律优先级准确率：75.1%（300条测试集）。

## 四、知识库设计

### 4.1 知识库结构

| 知识库 | 文件 | 用途 |
|--------|------|------|
| 📚知识库-PRD规范检查 | 📚知识库-PRD规范检查.md | PRD风险检查规则（场景错配、埋点违规、数据无来源等） |
| 📚知识库-反馈分类规则 | 📚知识库-反馈分类规则.md | 反馈分类规则+50+品类核心功能映射+few-shot示例 |

### 4.2 知识库在Agent中的作用

知识库通过Dify的knowledge-retrieval节点接入，LLM节点通过context配置引用知识库检索结果：

```yaml
context:
  enabled: true
  variable_selector:
  - '知识库节点ID'
  - result
```

当用户输入反馈时，知识库会检索最相关的分类规则和few-shot示例，作为LLM的上下文参考，确保分类一致性。

## 五、规则引擎兜底

LLM输出经过Code节点（反馈规则引擎）进行兜底处理：

1. **JSON解析**：从LLM输出中提取JSON（处理markdown代码块包裹）
2. **字段校验**：检查sentiment/type/priority是否在合法枚举值内
3. **一致性修正**：
   - 正向情绪 → 优先级强制为低
   - 建议类关键词 → 类型强制为体验
   - 功能失效关键词 → 优先级强制为高
4. **用户标签提取**：付费用户/新用户/老用户
5. **隐含需求提取**：从反馈中推断隐含需求

## 六、测试与评估

### 6.1 标准答案生成方法

采用**LLM共识法**：
1. 同一模型对每条反馈跑3轮分类
2. 多数投票决定标签（2/3或3/3一致）
3. 争议case（3轮全不同）→ 人工仲裁

### 6.2 评估指标

| 指标 | 定义 | 及格线 | 优秀线 |
|------|------|--------|--------|
| 情绪准确率 | 情绪正确数/总数 | ≥85% | ≥92% |
| 类型准确率 | 类型正确数/总数 | ≥80% | ≥90% |
| 优先级准确率 | 优先级正确数/总数 | ≥75% | ≥88% |
| 综合准确率 | 三项全对数/总数 | ≥70% | ≥85% |

### 6.3 当前测试结果

| 维度 | 准确率 |
|------|--------|
| 有效分类率 | 95%（100条真实差评） |
| 铁律拦截率 | 92.6%（88/95条走规则引擎，零LLM token消耗） |
| Agent vs 裸LLM情绪一致率 | 100% |
| Agent vs 裸LLM类型一致率 | 89.6% |
| Agent vs 裸LLM优先级一致率 | 79.2% |
| 情绪 | 98.3% |
| 类型 | 98.9% |
| 优先级 | 97.2% |
| 综合 | 95.0% |
| 铁律规则优先级 | 75.1%（纯规则引擎，300条测试集） |

> 铁律规则引擎：纯代码关键词匹配 + apply_priority_rules 四层后处理（缓解词降级 + 严重词提升 + 核心功能提升 + 中度词提升），准确率 75.1%。
