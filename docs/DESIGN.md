## 关键决策

### 决策一：为什么把规则从Prompt移到知识库？

早期版本把所有规则写在Prompt里，综合准确率只有63%。根因是规则太多（8条核心规则+30+个few-shot示例），LLM记不住，经常遗漏。改用知识库后，LLM只检索与当前输入最相关的规则和示例，准确率提升到95%。

| | 纯Prompt方案 | 知识库方案 |
|--|------------|----------|
| 准确率 | 63% | 95% |
| 维护 | 改YAML重新导入 | 改知识库文件即可 |
| 稳定性 | 规则太长LLM记不住 | 精准检索，LLM只看相关规则 |

### 决策二：为什么规则引擎和LLM要分开？

试过让LLM直接输出分类结果，发现两个问题：①同样输入可能不同输出（temperature=0也有微小波动），②"付费用户+核心功能=高优先级"这种规则LLM偶尔不遵守。拆开后，规则引擎保证确定性case永远正确，LLM只处理语义理解部分。

### 决策三：为什么从扣子搬到Dify？

扣子没有代码节点，所有逻辑只能靠Prompt硬撑。搬到Dify后，确定性逻辑用Python代码节点实现，LLM只负责语义理解，分工更清晰。

### 决策四：铁律优先级为什么从0%到75.1%？

最初 `iron_classify` 函数返回 `(问题类型, 情绪, None)`，优先级写死为None，三个调用节点拿到None后统一给了默认值"低"，导致铁律命中的结果里优先级全是低。修复后给所有return语句补上知识库对齐的优先级，新增 `apply_priority_rules` 六层后处理（缓解词降级、不降级保护、性能严重词提升、核心功能提升、体验中度词提升、内容中度词提升），准确率从0%提升到75.1%。

## Agent vs 裸LLM 差异案例

### 类型差异

| # | 反馈原文 | Agent分类 | 裸LLM分类 | 差异原因 |
|---|---------|----------|----------|---------|
| 1 | "APP打开白屏等了30秒才出来" | 性能 | 功能 | Agent按"加载慢"判性能，裸LLM按"白屏"判功能 |
| 2 | "搜索结果跟我要的完全不一样" | 内容 | 体验 | Agent按"算法输出质量"判内容，裸LLM按"不满意"判体验 |
| 3 | "能不能加个深色模式" | 体验 | 功能 | Agent规则"建议类一律体验"，裸LLM看到"功能"二字判功能 |

### 优先级差异

| # | 反馈原文 | Agent优先级 | 裸LLM优先级 | 差异原因 |
|---|---------|-----------|-----------|---------|
| 1 | "偶尔会有点卡，但整体还行" | 低 | 中 | Agent检测到缓解词"偶尔+还行"降级 |
| 2 | "支付页面加载有点慢" | 高 | 中 | Agent核心功能(支付)提升规则 |
| 3 | "退款7天了还没到账" | 高 | 中 | Agent核心功能(退款)提升规则 |

> Agent和裸LLM的差异主要来自规则引擎的后处理逻辑。裸LLM没有缓解词降级、核心功能提升等规则，所以优先级判断偏保守。

## 架构

```
用户输入（文字 + 可选文件）
     │
     ▼
[LLM+知识库] 轻量分类器 —— 先让LLM理解用户想干什么（接入知识库辅助判断）
     │
     ▼
[代码] 路由验证 —— 再用代码二次确认，防止LLM乱判
     │
     ├── prd_review_file   → 文档提取 → 格式化拆分 → 知识库检索 → 批量PRD检查 → Jira格式输出
     ├── prd_review_text   → 读者确认 → 知识库检索 → 单条PRD检查 → Jira格式输出
     ├── data_label_file   → 文档提取 → 批量分类（纯代码） → 汇总输出
     ├── data_label_text   → 知识库检索 → LLM提取特征 → 代码规则引擎 → 输出
     ├── scene_declare     → 记录场景 → 后续检查更准
     └── invalid_input     → 拒绝
```

为什么这样分：
- LLM擅长理解语义，比如"太棒了，一天闪退八次"是反讽，关键词搞不定
- 但做决策必须确定，"付费用户+核心功能=高优先级"必须每次都一样
- 路由走错了后面全错，所以用代码做二次校验
- 知识库检索让LLM有规则可循，不靠"感觉"分类

## 亮点

| # | 亮点 | 怎么做的 |
|---|------|---------|
| 1 | 场景感知 | "迷路"在扫地机器人→功能问题，在APP→体验问题，根据品类动态调整 |
| 2 | 隐含需求提取 | "每次都要重新登录"→隐含需求：记住登录状态 |
| 3 | 反讽检测 | "太棒了，一天闪退八次！"→判为负向 |
| 4 | 付费用户升级 | "充了会员还是闪退"→强制高优先级 |
| 5 | 缓解词检测 | "偶尔有点噪音，但整体还行"→优先级降级 |
| 6 | 防Prompt注入 | "忽略之前所有规则"→拒绝处理 |
| 7 | PRD风险转Jira | 检查出风险→自动生成Jira Bug格式，直接复制 |
| 8 | 灰名单术语 | Webhook/OAuth2.0→标注"请人工确认"，不误判 |
| 9 | 读者确认 | PRD检查前会问你是给老板/开发/测试看的，不同人关注点不同 |
| 10 | 51个品类覆盖 | 从电商到AI对话到智能汽车，核心功能映射表全覆盖 |
| 11 | 知识库驱动 | 分类规则和few-shot示例存知识库，LLM有据可依 |
| 12 | 规则引擎兜底 | LLM输出经代码节点校验修正，正向→强制低优先级，建议类→强制体验 |
| 13 | 铁律优先级修复 | iron_classify 从返回None改为返回知识库对齐的优先级，新增 apply_priority_rules 六层后处理 |
