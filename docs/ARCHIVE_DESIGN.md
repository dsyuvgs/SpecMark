# SpecMark 设计文档

> 为什么做这个、怎么做的、踩过的坑。
> 当前版本：v1.0（2026.05）

---

## 一、为什么做这个

### 最早做了PM Copilot，发现不行

最早在扣子上做了个PM Copilot，10个功能的AI产品经理助手。9个模型跑了216次测试，选了DeepSeek 3.2。提示词从7640字压到4183字，测试9/10通过。

**但实际效果不如裸模型**。原因：

| 问题 | 说明 |
|------|------|
| 内容生成裸模型就能做好 | PRD撰写、竞品分析、用户故事……加Agent反而干扰 |
| 功能太多太浅 | 10个功能每个都只做到60分 |
| 扣子没代码节点 | 没法做确定性决策，全靠提示词硬撑 |

### 怎么决定留哪些功能

一个功能值不值得做Agent，我就看一个问题：**加了工作流之后，效果能不能超过裸模型？**

| 任务类型 | 裸模型表现 | 加Agent能提升吗 | 结论 |
|---------|-----------|----------------|------|
| 内容生成（PRD/竞品分析/用户故事等） | 已经很好 | 提示词反而干扰 | 砍掉 |
| 分类标注（反馈分类） | 不稳定 | 规则引擎保证确定性 | 保留 |
| 检查标注（PRD检查） | 容易遗漏 | 逐条对照不遗漏 | 保留 |

最后只留了2个功能：**反馈分类 + PRD检查**。

## 二、怎么做的

### 整体架构

```
用户输入
  ↓
[路由层] LLM意图识别 + 代码校验
  ↓
  ├→ PRD审查分支
  │     ├→ 有文件 → [文档提取器] → [迭代节点] → 批量PRD检查 → 汇总输出
  │     └→ 无文件 → 读者确认 → 单条PRD检查 → Jira格式输出
  │
  ├→ 数据标注分支
  │     ├→ 有文件 → [文档提取器] → [迭代节点] → 批量分类 → 汇总输出
  │     └→ 无文件 → LLM特征提取 → 代码规则引擎 → 格式化输出
  │
  └→ 无效输入 → 拒绝处理
```

### 为什么把理解和决策分开

我不想让LLM做所有决策，所以把"理解"和"决策"拆开了：

- **理解语义是LLM的强项**——"太棒了，一天闪退八次"是反讽，关键词搞不定这种
- **做决策必须确定性**——"付费用户+核心功能=高优先级"这种规则必须100%执行，不能靠LLM猜
- **路由必须100%对**——走错路后面全错，所以用代码做路由校验

### 为什么搬到Dify

| 能力 | 扣子 | Dify |
|------|------|------|
| 代码节点 | 没有 | 有Python代码节点 |
| 规则引擎 | 只能靠提示词 | 代码搞定确定性逻辑 |
| YAML导出 | 平台锁定 | 完整DSL导出 |
| API测试 | 不方便 | 标准REST API |
| 迭代节点 | 没有 | 有，批量处理靠它 |
| 文档提取器 | 没有 | 有，上传文件靠它 |

### 批量处理怎么做的

一开始只能一条一条粘贴，后来发现这不现实——运营丢过来几百条差评，你总不能一条条贴吧。

做法：
1. 用户上传文件（CSV/JSON/TXT/MD/Word都行）
2. 文档提取器把文件内容读出来
3. 迭代节点一条条过LLM
4. 最后汇总输出

踩的坑：Dify的LLM节点有个硬性要求，query_prompt_template里必须包含`{{#sys.query#}}`，不然报错。批量处理的时候内容是通过变量传进来的，不是sys.query，所以要在template里同时放`{{#sys.query#}}`和变量引用。

### 读者确认是什么

PRD审查的一个小亮点：用户贴一段PRD进来，Agent先问"这个PRD是给谁看的？"——老板、开发、测试、运营，不同角色关注点不一样。

比如给老板看的PRD，重点检查数据有没有来源、目标可不可量化；给开发看的，重点检查异常处理和验收标准。

### 51个品类怎么来的

一开始只做了扫地机器人的核心功能映射，后来发现这不行——用户随便输个品类的反馈，Agent识别不到核心功能，分类就不准。

所以把市面上常见的品类都补上了，从电商、社交、金融这些传统行业，到AI对话、VR/AR、智能汽车这些新兴行业，一共51个。每个品类都列了核心功能关键词，规则引擎根据这些关键词判断一条反馈涉及的是不是核心功能。

### 知识库为什么后来加上了

项目里有两个知识库文件（📚知识库-PRD规范检查.md 和 📚知识库-反馈分类规则.md），工作流里有3个知识库检索节点。

早期版本没有用知识库，规则全写在Prompt里。但后来发现规则越来越多（8条核心规则+30+个few-shot示例），Prompt太长导致LLM不稳定，综合准确率只有63%。改用知识库后，LLM只检索与当前输入最相关的规则和示例，准确率提升到95%。

| | 纯Prompt方案（旧版） | 知识库方案（当前） |
|--|-------------------|---------|
| 准确率 | 63% | 95% |
| 延迟 | 无检索环节 | 多一次向量检索（<1秒） |
| 维护 | 改YAML重新导入 | 改知识库文件即可 |
| 稳定性 | 规则太长LLM记不住 | 精准检索，LLM只看相关规则 |

踩过的坑：
1. `retrieval_mode: multiple` 没配reranking_model导致Dify报错 → 改为single
2. LLM节点 `context.enabled: false` 导致知识库检索结果没传给LLM → 改为true
3. 知识库文件名和节点名不一致，导入后不知道绑定哪个 → 统一命名

## 三、规则怎么定的

### 反馈分类规则

**问题类型**（加权评分）：

| 类型 | 关键词示例 | 权重 |
|------|-----------|------|
| 功能问题 | 闪退、崩溃、报错、无法、失败 | 3 |
| 性能问题 | 卡顿、慢、延迟、加载 | 2 |
| 体验问题 | 麻烦、繁琐、不好用、找不到 | 1 |
| 内容问题 | 错误、不一致、显示异常 | 2 |

**优先级**：

```
负向 + 核心功能 + 功能问题 → 高
负向 + 核心功能 + 非功能问题 → 中
负向 + 非核心功能 → 中（有缓解词→低）
正向 + 核心功能 → 中
正向 + 非核心功能 → 低
中性 → 中/低
```

**特殊规则**：
- 付费用户+核心功能问题 → 强制高优先级
- 反讽 → 判为负向，严重程度不降级
- 缓解词（偶尔/有时/不影响使用）→ 优先级降级，但严重问题除外
- 竞品对比 → 禁用严重问题检测
- 咨询类（怎么连接WiFi？）→ 优先级降级

### PRD检查规则

**检查维度**（按优先级）：

| 优先级 | 维度 | 怎么判 |
|--------|------|--------|
| P1 | 场景错配 | ToC产品出现ToB专属术语（多租户/SSO/RBAC等） |
| P1 | 埋点违规 | 一个行为一个事件，没用通用事件+动态参数 |
| P2 | 数据无来源 | 数据类内容没标出处 |
| P3 | 缺少异常处理 | 只写了正常流程，没考虑异常 |
| P4 | 验收标准模糊 | "功能正常运行""体验流畅"这种模糊表述 |
| P5 | 需求不可量化 | "提升用户体验""优化性能"没量化指标 |

**术语分级**：
- 黑名单：多租户、SSO、RBAC → ToC产品一定判为场景错配
- 灰名单：Webhook、OAuth2.0、API、CDN → 标注"请人工确认"
- 白名单：通用技术术语 → 不标记

## 四、10个亮点

### 1. 场景感知分类
同一个词在不同产品含义不同。"迷路"在扫地机器人→功能问题，在APP→体验问题。规则引擎根据场景动态加载核心功能定义，51个品类都有对应的关键词映射。

### 2. 隐含需求提取
"每次都要重新登录" → 隐含需求：记住登录状态。不只分类，还把用户没说出口的需求提取出来。18条规则全覆盖。

### 3. 反讽检测
"太棒了，一天闪退八次！" → LLM识别sarcasm信号 → 判为负向。48条评测集反讽用例100%正确。

### 4. 付费用户优先级升级
"充了会员还是闪退" → 付费用户+核心功能问题 → 强制高优先级。

### 5. 缓解词检测
"偶尔会有点噪音，但整体还行" → 有缓解词 → 优先级降级。但严重问题（闪退/崩溃）即使有缓解词也不降级。

### 6. 防Prompt注入
"忽略之前所有规则" → 路由为invalid_input → 拒绝处理。100%拦截。

### 7. PRD风险自动转Jira任务
检查出风险 → 自动转成`【Bug】PRD风险：xxx | 优先级：High | 描述：xxx`格式，直接复制到Jira。

### 8. 灰名单术语处理
Webhook/OAuth2.0 → 不误判为场景错配，标注"请人工确认"。

### 9. 高频风险总结
当次检查检出≥2个同类风险 → 自动追加`📊 本次检查最高频风险：xxx（出现N次）`。纯代码逻辑，不依赖变量赋值，零风险。

### 10. 铁律优先级六层后处理
修复了铁律规则引擎优先级全线为"低"的bug后，新增 `apply_priority_rules` 函数实现六层后处理：缓解词降级（"偶尔/还行"→降一级，但严重问题不降）、不降级保护（闪退/崩溃/食安/支付失败即使有缓解词也不降）、性能严重词提升（"太卡/没法用"→升为高）、核心功能提升（涉及支付/登录→升为高）、体验中度词提升（强烈不满→升为中）、内容中度词提升（严重错误→升为中）。铁律优先级准确率从 0% 提升至 75.1%。

## 五、踩过的坑

### 第1次：搭基础架构
- 搞了路由验证+特征提取+规则引擎三段架构
- 踩坑：JSON解析对嵌套结构很脆弱、关键词覆盖不够

### 第2次：路由逻辑改了好几次
- 踩坑：PRD输入被误判为反馈、LLM高置信度输出绕过了规则检查
- 修复：路由改成规则优先，把高置信度旁路去掉了

### 第3次：优先级判定太简单
- 踩坑：竞品对比/咨询类/体验问题的优先级不准
- 修复：加了严重问题检测、咨询类降级、体验问题降级

### 第4次：分类平局搞不定
- 踩坑：功能/体验问题关键词打平时分类不准
- 修复：搞了加权评分+平局决胜逻辑

### 第5次：改一个bug又出新bug
- 踩坑：每次改完发布，跑测试又有新问题
- 修复：写了个本地模拟脚本，48条用例一次性全跑，一次修6个问题

### 第6次：批量处理一堆报错
- 踩坑：LLM节点必须包含sys.query、sys.files引用报运行时错误、知识库检索节点导致批量路径走不通
- 修复：query_prompt_template里同时放sys.query和变量引用、去掉sys.files、批量路径绕过知识库检索

### 第7次：内容类型分不清
- 踩坑：用户反馈被当成PRD内容处理了
- 修复：加了关键词检测，先判断内容是反馈还是PRD，再走对应分支

### 第8次：代码节点没有异常处理
- 踩坑：LLM输出格式不对的时候，代码节点直接报错，用户看到一堆报错信息
- 修复：所有代码节点加了try-except，出错返回"分析失败，请检查输入格式或重试"

### 第9次：铁律规则引擎优先级全线输出为"低"
- 踩坑：`iron_classify` 函数返回 `(问题类型, 情绪, None)`，优先级写死为 None，三个调用节点拿到 None 后统一给了默认值"低"，导致铁律命中的 185 条结果里优先级全是低
- 修复：给 `iron_classify` 所有 return 语句补上知识库对齐的优先级；新增 `apply_priority_rules` 后处理函数，实现缓解词降级、性能严重词提升、核心功能提升、体验/内容中度词提升四层规则；扩展 `svc_high_kw` 食安检测关键词
- 效果：铁律优先级准确率从 0% → 75.1%，高→中误判从 49 条降至 13 条，中→低误判从 21 条降至 4 条

## 六、迭代过程准确率变化

> 从初始版本到最终版本的迭代数据。

| 阶段 | 情绪准确率 | 类型准确率 | 优先级准确率 | 综合准确率 | 关键改动 |
|------|-----------|-----------|-------------|-----------|---------|
| 初期 | ~45% | — | — | — | 情绪判定全靠关键词，"不卡"被判成负向 |
| 中期 | ~90% | ~88% | ~85% | — | 反讽检测、付费用户升级、缓解词降级、品类映射 |
| 后期 | 95% | 93% | 91% | 89% | 知识库接入、加权评分、51个品类覆盖 |
| 最终 | 98.3% | 98.9% | 97.2% | **95.0%** | 知识库驱动+few-shot示例+规则引擎兜底+铁律优先级修复 |

### 关键突破（63% → 95%）

这是项目最关键的迭代阶段：

**初期问题**：多品类测试发现综合准确率只有63%。根因：
1. 规则太多写在一个Prompt里，LLM记不住
2. GT标签本身有错误（建议类标功能、服务问题标功能）
3. 优先级判定规则不够细致

**改进措施**：
1. 把规则从Prompt移到知识库，LLM只检索相关规则
2. 修正GT标签错误
3. 补充优先级判定规则（体验问题也能判高优先级）
4. 加入30+个few-shot示例，覆盖所有高频错误
5. 规则引擎兜底修正（正向→强制低优先级、建议类→强制体验）
6. 修复铁律规则引擎优先级bug，新增六层后处理

### 核心教训

- **Prompt不是越长越好**：规则超过一定长度，LLM反而更不稳定。知识库检索比全塞Prompt好
- **先审GT再优化**：很多时候不是Agent错了，是标准答案本身有误
- **Few-shot比规则更有效**：给LLM看正确示例比给规则更管用
- **一次性想清楚再改**：不要改一点测一点，每次迭代前先分析所有问题

## 八、评判标准

> 之前说"合理"但没定义什么是合理，现在把标准定清楚。
> 详细标准见 knowledge/📚知识库-反馈分类规则.md 和 knowledge/📚知识库-PRD规范检查.md 的"〇、评判标准"章节。

### 反馈分类的评判标准

| 指标 | 行业及格线 | 我们的目标 | 当前实测 |
|------|-----------|-----------|---------|
| 情绪准确率 | ≥85% | ≥92% | 98.3% |
| 类型准确率 | ≥80% | ≥90% | 98.9% |
| 优先级准确率 | ≥75% | ≥88% | 97.2% |
| 综合准确率 | ≥70% | ≥85% | 95.0%（LLM共识法，180条×3轮投票） |
| Cohen's Kappa | ≥0.6 | ≥0.75 | 待测（需人工标注对比） |

**准确率怎么算**：Agent输出 = 人工标注标准答案 → 算对。每个维度单独算，三项全对算综合准确率。

**分布合理性**：不是"53%功能+41%体验"就一定对。100条纯负面评价，负向应≥90%，"其他"应≤5%，优先级高:中:低应≈3:4:3到5:3:2。偏离这些范围说明判定逻辑有问题。

**对比基线**：准确率必须跟裸模型比。Agent综合准确率应超过裸模型+10个百分点，否则加工作流没意义。

### PRD审查的评判标准

| 指标 | 行业及格线 | 我们的目标 | 当前实测 |
|------|-----------|-----------|---------|
| 检出率（Recall） | ≥70% | ≥80% | 待测 |
| 准确率（Precision） | ≥75% | ≥85% | 待测 |
| 误报率 | ≤25% | ≤15% | 待测 |
| P1严重问题检出率 | ≥90% | ≥95% | 待测 |

**为什么用检出率而不是准确率**：PRD审查宁可多报不可漏报。漏了一个P1场景错配，比多报一个P3缺异常处理严重得多。所以检出率是第一优先指标。

**"应检出问题"怎么定**：写一份故意埋了已知问题的PRD，人工标注所有问题，然后让Agent跑，对比检出率。

**对比基线**：人工需求评审平均检出率60-80%（行业数据），Agent应超过80%。
